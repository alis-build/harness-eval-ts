# Schema versioning

The `schemaVersion` field is a constant `"1.0"` in the current release. Future breaking changes will increment the major version (e.g., `"2.0"`). Non-breaking additions will keep the same version.

Consumers SHOULD check `schemaVersion` before parsing. If the version is unknown, attempt best-effort consumption rather than rejecting.

# Database storage pattern

The envelope is designed for direct JSON column storage in most databases:

```sql
CREATE TABLE eval_runs (
  run_id UUID PRIMARY KEY,
  started_at TIMESTAMPTZ NOT NULL,
  schema_version TEXT NOT NULL,
  behavioral_pass BOOLEAN NOT NULL,
  outcome_pass BOOLEAN,
  envelope JSONB NOT NULL
);

INSERT INTO eval_runs (run_id, started_at, schema_version, behavioral_pass, outcome_pass, envelope)
VALUES (
  $1::uuid,
  $2::timestamptz,
  $3,
  ($4::jsonb -> 'summary' ->> 'behavioralPass')::boolean,
  ($4::jsonb -> 'summary' ->> 'outcomePass')::boolean,
  $4
);
```

Key fields to index for dashboards: `started_at`, `behavioral_pass`, `provenance.git.commit`.
